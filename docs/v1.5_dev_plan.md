## 0. 总体策略

### 0.1 拆分逻辑

V1.5 PRD 共 **4 个模块、22 个子需求**（SES-01~09、KB-01~06、FILE-01~05、TASK-01~05）。直接按模块切并行开发会陷入"会话标题异步生成依赖 Celery、Celery 又依赖 Redis、Redis 又没人初始化"的循环卡点。

按**技术依赖**重排为 6 个阶段（**S0~S5**）：

```
S0 基础设施 ─┬─> S1 会话 CRUD ─┐
            ├─> S2 知识库 CRUD ─> S3 文件入库 ─> S4 异步标题/摘要 ─> S5 联调
            └─> （Celery+Redis）
```

S0 完成后，S1 / S2 可并行（互不依赖）；S3 必须等 S2；S4 依赖 S1+S0；S5 把所有改动接入 `/chat/stream` 并端到端联调。

### 0.2 工程原则

1. **不破坏 V1.0 核心契约**：V1.0 测试套件（62 用例）经包装层改造后必须保持绿；Agent ↔ Service 之间的 `run_stream()` 接口契约不动。
2. **小步快跑**：每个子需求独立提交，commit message 带 `[V1.5][SES-01]` 这样的前缀。
3. **数据库可清空**（用户已确认）：V1.5 阶段 PG 库可重置；改 schema 不写迁移脚本，靠 lifespan `Base.metadata.create_all` + 文档说明"首次启动前清库"。
4. **Milvus 数据隔离**：V1.0 唯一 Collection `knowledge_chunks` 同样**可清空**；V1.5 新增 KB 一律走动态命名 `kb_{kb_id_no_hyphen}`，无 V1.0 历史数据迁移负担。
5. **统一响应格式覆盖全量**：V1.0 老接口（`POST /sessions` / SSE 报文等）也包成 `{code, message, data}`；V1.0 测试同步改。SSE 报文本身已经是结构化 JSON，不再二次包装（保持 V1.0 SSE 协议字段 `event/type/...` 不变）。
6. **代码注释 / 文档 / commit message 一律简体中文**。

### 0.3 已确认的关键决策（2026-06-11）

| 决策点 | 选择 | 说明 |
|---|---|---|
| 阶段切分 | 6 阶段（S0~S5），独立验收 | 出问题易定位回滚 |
| `.doc` 格式 | 推迟到下一迭代 | V1.5 仅支持 .pdf / .docx / .md / .txt；上传 .doc 返 415 + 提示"请另存为 docx" |
| DB 迁移 | 不做迁移，库可清空 | `Base.metadata.create_all` 直接覆盖；启动文档加一段"首次启动须清空 PG / 删除 V1.0 Milvus Collection" |
| 统一响应格式 | V1.0 + V1.5 全覆盖 | V1.0 老接口与单测同步改 |
| Celery 池 | 开发 `--pool=solo` / Linux 生产 `prefork` | 写到 `docs/celery_dev_guide.md` |
| 架构文档 | 追加到 `docs/architecture.md` | 标题改通用，每完成一阶段同步增章节，不预先写 |
| NER 模型 | 配置层支持独立切换 + 多份配置预留 | `.env.example` 预留 `KG_NER_MODEL` 与若干备选注释行，后期可一键切换做模型对比 |
| 验收方式 | 模式 A：Claude 给命令，用户跑后贴输出 | 长进程（worker / uvicorn / smoke）一律不自动启 |

### 0.4 用户操作约定（沿用 CLAUDE.md）

Claude **不主动执行**：
- `uv pip install ...`（只更新 `requirements.txt`）
- `docker run / docker compose up`（只写 `docker-compose/docker-compose.yml`）
- `pytest`、`uvicorn`、`celery worker`、联调脚本（只给命令，用户跑完贴回结果）

| 子需求 | 方法 + 路径 | 实现位置 |

| SES-05 | `DELETE /api/v1/sessions/{session_id}`（级联删消息） | 同上 |
| SES-06 | `GET /api/v1/sessions/{session_id}/messages`（游标翻页） | 新增 endpoint |
| SES-09 | `/chat/stream` 历史消息窗口截断（最近 N 条，system 必含，按 `CONTEXT_WINDOW_MESSAGES`） | 改 [app/services/chat_service.py](../app/services/chat_service.py) `_load_history` |

### 技术要点

1. **统一响应格式**（已确认 D4 全覆盖）：PRD §7.1 要求 `{code, message, data}` 包裹。**V1.0 老接口与新接口一律包装**，封装位置：
   - 新建 `app/schemas/response.py::ApiResponse[T]` 通用泛型容器
   - 新建 `app/api/exceptions.py` 统一异常处理器，把 `HTTPException` / 校验失败 / 未捕获 5xx 全部翻译成 `{code, message, data: null}`
   - V1.0 的 `tests/test_sessions_api.py` 等单测需同步改 assert，取 `data.id` 而不是 `id`
   - SSE 报文（`/chat/stream`）**不再二次包装**，沿用 V1.0 协议（`event: message|control`, `data: {event, type, ...}`），但 SSE 之外的错误路径（404 等）也走统一格式
2. **message_count 维护**：写消息时同步 `+=1`；删消息（SES-05）时整张删，不必逐条减。
3. **游标翻页（SES-06）**：`limit` + `before`（消息 ID）。SQL：`WHERE session_id=? AND created_at < (SELECT created_at FROM chat_messages WHERE id=?) ORDER BY created_at ASC LIMIT ?`。
4. **错误码**：404 用 PRD §7.2 业务 code `40400`，由 `app/api/exceptions.py` 统一拦截 `HTTPException` 并包装。

## S2 · 知识库 CRUD + Milvus 多 Collection 管理

**目标**：PRD §3.2 全量落地。**KB-06（KB 关联对话）放 S5**，因为依赖 S3 的文件入库才能验证。

### 子需求映射

| 子需求 | 实现位置 |
|---|---|
| KB-01 创建 + 同步初始化 Milvus Collection | `app/api/v1/endpoints/knowledge_bases.py`（新文件）+ `app/services/kb_service.py`（新文件）+ `app/rag/milvus_client.py` 增 `create_kb_collection(kb_id)` |
| KB-02 列表查询（分页 + 统计） | 同上 |
| KB-03 详情查询（含 `entity_count`，查 Neo4j 计数） | 同上 + `app/kg/query.py` 增 `count_entities_by_kb(kb_id)` |
| KB-04 元数据更新（仅 name / description） | 同上 |
| KB-05 删除（**严格按 Milvus → PG → Neo4j 顺序**；任一步失败要回滚） | 同上 |

### 技术要点

1. **Collection 命名**：`f"kb_{kb_id.hex}"`（去 hyphen，Milvus 不允许 `-`）。封装成 `app/rag/naming.py::build_kb_collection_name(kb_id)`，整个项目只此一处。
2. **Milvus Schema 扩展**：新增 `kb_id` 字段（VARCHAR 64）写入冗余值，便于跨 Collection 调试（PRD §5.4）。
3. **KB-01 同步建库**：`create_collection → create_index → load`。失败需回滚 PG（用 SQLAlchemy 事务）。
4. **KB-03 entity_count 性能**：每次详情查询都 `MATCH (n:Entity {kb_id}) RETURN count(n)`，OK；列表查询不查 Neo4j（PRD 明确说从冗余字段读）。
5. **KB-05 删除**：用 try/except 包三步，任一失败 raise 500；Milvus drop 是真正不可恢复的，**先 drop Milvus、再删 PG、再删 Neo4j**（PRD §3.2 KB-05 给的就是这个顺序）。
6. **Neo4j 节点写入 `kb_id`**：PRD §5.5 要求所有 `Entity` / `Document` 节点新增 `kb_id` 属性。S2 阶段先**修 writer 接受 kb_id 参数**，**写入逻辑 S3 文件入库阶段才真正用到**。

### 测试

- `tests/test_kb_api.py`（mock Milvus client）
- `tests/test_kb_service.py`（含 KB-05 回滚分支）

### 验收

- 创建 KB 后 Milvus 中出现 `kb_xxx` Collection 且 loaded
- 删除 KB 后 Milvus / PG / Neo4j 三库均无残留
- name 冲突返回 409 + business code `40900`

## S3 · 文件上传 + 异步入库（V1.5 核心）

**目标**：PRD §3.3 + §3.4 TASK-02/TASK-03。这是 V1.5 工程量最大的一段。

### S3.1 文件上传同步接口（FILE-01）

- `app/api/v1/endpoints/kb_files.py`（新文件）
- `POST /api/v1/knowledge-bases/{kb_id}/files`：
  - `multipart/form-data` 接收 `file`
  - 校验：KB 存在 / 大小 ≤ `MAX_FILE_SIZE_MB` / MIME 在允许集
  - 保存到 `{UPLOAD_DIR}/{kb_id}/{file_id}/{filename}`
  - 写 `kb_files` 行（`status=pending`、`progress=0`）
  - **触发 Celery 任务**：`parse_and_ingest_task.delay(file_id, kb_id)`，把 task_id 写回 `celery_task_id`
  - 返回 `file_id` + 当前状态，**不等入库完成**
- 错误码：404 / 413 / 415（PRD §7.2）

### S3.2 文件查询/删除/重新入库（FILE-02 ~ FILE-05）

- `GET /knowledge-bases/{kb_id}/files`（分页）
- `GET /knowledge-bases/{kb_id}/files/{file_id}`（含 progress、error_message、celery_task_id）
- `DELETE /knowledge-bases/{kb_id}/files/{file_id}`：
  - 若 `status=processing`：先 `celery_app.control.revoke(task_id, terminate=True)`
  - 顺序：Milvus（按 `document_id == file_id` 过滤删）→ Neo4j（`MATCH (n {kb_id, document_id}) DETACH DELETE n`）→ PG → 磁盘
- `POST /knowledge-bases/{kb_id}/files/{file_id}/reindex`：
  - 复用上一步前 3 个清理步骤（**保留磁盘文件**）
  - `status` 重置 `pending`、`progress=0`
  - 重新触发任务

### S3.3 解析器层（PRD §6.1）

- `app/ingest/parser.py`（新文件）：按 MIME 分发到具体解析函数
  - `.pdf` → PyMuPDF `fitz.open()`，逐页 `page.get_text()`；空文本页记 warning、跳过
  - `.docx` → `python-docx`，提取正文段 + 表格、忽略页眉脚/图
  - `.doc` → 调 `soffice --headless --convert-to docx`，再走 docx 分支（**Windows 上 LibreOffice 路径需配置项**，可选 V1.5 简化为只支持 .docx）
  - `.md` → 去 Markdown 符号（用 `markdown-it-py` 解析后取纯文本，依赖加入）
  - `.txt` → `open(..., encoding="utf-8")`，UnicodeDecodeError → 抛 415

> 决策：**V1.5 第一版先支持 .pdf / .docx / .md / .txt**，`.doc` 留为下一迭代（LibreOffice 在 CI/Windows 下踩坑成本高）。

### S3.4 切片层

- `app/ingest/splitter.py`：包装 `RecursiveCharacterTextSplitter`，按 KB 的 `chunk_size / chunk_overlap` 切（token 长度估算用 tiktoken `cl100k_base`，中文场景 ≈ 1.5 char/token）
- 输出 `Chunk(text, chunk_index)` 列表

### S3.5 Celery 任务（TASK-02 / TASK-03）

- `app/tasks/ingest_task.py`：`parse_and_ingest_task(file_id, kb_id)`
- 严格按 PRD §3.4 TASK-02 七步走，每步完成后 update progress（用一个 helper `_update_progress(file_id, progress, status=None)`）
- **同步上下文**：Celery worker 是同步进程，要调 async 函数（embedding / Milvus / Neo4j）需用 `asyncio.run` 或 `nest_asyncio`。**选 `asyncio.run`**（每个任务独立事件循环，避免污染）
- **重试**（TASK-03）：装饰器配置 `autoretry_for=(MilvusException, RedisConnectionError)`、`max_retries=3`、`countdown=30`、`retry_backoff=True`
- **NER 软失败**（PRD §6.3）：try/except 包 NER 步骤，失败 → warning + `entity_count` 保持当前值 + 继续后续步骤
- **维度校验**（PRD §6.2 / 7.2）：embedding 返回维度 ≠ KB.embedding_dim → 任务 failed，error_message="向量维度不匹配，期望 4096 得到 ..."（HTTP 投影为 42200）

### S3.6 测试

- `tests/test_parser.py`：每种格式给一个小样本（`tests/fixtures/`）
- `tests/test_splitter.py`：边界条件（空文本 / 单 chunk / chunk_overlap > size 校验）
- `tests/test_ingest_task.py`：mock 掉 Milvus / Neo4j / embedding，验 progress 流转、异常路径、NER 软失败
- `tests/test_kb_files_api.py`：上传、查询、删除、重新入库

### 验收

- 用户手动跑：起 worker → 上传一份 PDF → 轮询 `GET /files/{id}` → progress 0 → 20 → 35 → 60 → 80 → 95 → 100
- Milvus 中 `kb_xxx` Collection 出现新切片
- Neo4j 中 `Entity` 节点带 `kb_id` 属性
- 删 KB 时所有该 KB 的切片、实体、文件、磁盘文件全部清理

## S4 · 会话标题与摘要异步生成

**目标**：PRD §3.1 SES-07 / SES-08 + §3.4 TASK-04 / TASK-05。

### 子需求映射

| 子需求 | 实现位置 |
|---|---|
| SES-07 / TASK-04 自动标题生成 | `app/tasks/session_task.py::generate_session_title_task`；触发点写在 `chat_service.stream_chat` 流末尾 |
| SES-08 / TASK-05 摘要生成 | `app/tasks/session_task.py::generate_session_summary_task`；`POST /api/v1/sessions/{id}/summarize` endpoint 触发，立即返 202 |

### 技术要点

1. **触发条件（SES-07）**：流式回答完成后判断 `session.title is None AND session.message_count == 2`（首次 user + 首次 assistant），满足才入队。重复触发要幂等（任务里再判一次）。
2. **Prompt 严格按 PRD §3.1 SES-07 给的模板**，输出做 `.strip().replace('"', '')` 清洗，截 20 字。
3. **摘要 SES-08**：endpoint 返 `{"code": 0, "data": {"task_id": "..."}}` + HTTP 202。任务取**全量** messages 拼接，调 LLM，结果写回 `summary` 和 `summarized_at`。
4. **任务失败不写空值**：try/except 包整段，except 分支只记日志、不 update `summary` 字段。

### 测试

- `tests/test_session_tasks.py`：mock LLM，验首次/重复触发逻辑、长度截断、失败不写空值

### 验收

- 起 worker，发首条消息 → 10s 内 GET session 看到 title 自动填充
- 调 `/summarize` → 立即返 202 + task_id → 短时间后 GET session 看到 summary 非空

## S5 · KB 关联对话 + 端到端联调

**目标**：把 KB-06 接进 `/chat/stream`，跑通完整 V1.5 业务流。

### 子需求映射

| 子需求 | 实现位置 |
|---|---|
| KB-06 `kb_ids` 参数 + 多 Collection 路由 + Neo4j `kb_id` 过滤 | 改 `app/schemas/chat.py::ChatRequest` 增 `kb_ids` 字段；改 `app/rag/retriever.py::search_knowledge_base` 接受 `kb_ids` 入参并跨 Collection 合并；改 `app/kg/query.py` 追加 `WHERE n.kb_id IN $kb_ids` 过滤 |

### 技术要点

1. **跨 Collection 合并**：`kb_ids=[A,B]` 时，依次 `MilvusClient.search(collection_name=kb_A_collection, ...)` + B，应用层按 score 重排序取 top_k。
2. **缺省行为**：`kb_ids` 未传 → 查全部 active KB（性能差，日志 warning）；传空 list → 不查 RAG（让 LLM 直接答）。
3. **SSE tool_start 必须携带 kb_ids**：在 retriever 的入参里塞 `kb_ids`，runner 序列化 args 时自然带上。

### 端到端联调脚本

`scripts/v1_5_smoke.py`：

1. 建 session
2. 建两个 KB（"气象库"、"应急库"）
3. 各传一份 PDF，轮询直到 progress=100
4. 用 `kb_ids=[气象库]` 提问 → 验返回只含气象库内容
5. 用 `kb_ids=[气象库, 应急库]` 提问 → 验合并结果
6. 触发 summarize → 短轮询拿到 summary
7. 删 KB → 验三库清理干净

## 风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| Windows 下 Celery 多进程并发不稳 | worker 起不来或任务丢 | `--pool=solo` 单进程跑（PRD 没要求高并发）；写到 `docs/celery_dev_guide.md` |
| LiteLLM 长文本摘要超 context | SES-08 报错 | summary 任务前先做长度判断，超 32k token 改为 map-reduce 分段摘要 |
| Milvus 多 Collection 资源占用 | 内存暴涨 | KB-05 删除时确保真 drop；运维文档提示单实例 KB 数量上限（建议 ≤ 50） |
| PostgreSQL ALTER 在带数据老库上失败 | 启动崩溃 | 已规避：V1.5 库可清空，走 `create_all`；启动前清库一次 |
| V1.0 测试因统一响应格式改造而批量失败 | 阶段验收受阻 | S1 第一步先把 `ApiResponse` + 异常处理器落地，跟 V1.0 测试改造同步 commit；任何阶段先跑全量 pytest |
| NER 大量 chunk 烧 LLM token | 入库慢 + 贵 | TASK-02 NER 步骤按批并发但限速（asyncio.Semaphore=4）；KG_NER_MODEL 用 flash 模型 |
| 重新入库（FILE-05）旧 chunk 残留 | 检索污染 | 删除 Milvus 时按 `document_id == file_id` 过滤，不仅删 chunk_id 区间 |
